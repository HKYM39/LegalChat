## 背景

项目当前的核心能力集中在在线检索与回答，但 `Data/Source/` 下的原始 PDF 仍未进入 PostgreSQL。根据 `Doc/Backend.md` 与 `Doc/ProjectPRD.md`，Python 的职责只限于离线 PDF 处理与标准化输出，在线服务仍必须由 Hono + TypeScript 提供。因此，本次设计需要在不破坏现有在线架构边界的前提下，建立一条从 PDF 到数据库记录的离线流水线，并与既有 Drizzle schema 对齐。

现有样本文件为澳洲高院判例 PDF，文件名中已经包含案名、引用和日期信息，但 PDF 内部排版可能存在页眉页脚、换行、段落编号不规整等问题。离线流程必须优先保证结构保留、标识稳定与可追溯性，而不是只追求一次性文本抽取。

## 目标 / 非目标

**目标：**

- 建立可重复执行的离线 Python 流程，批量读取 `Data/Source/` 下的 PDF。
- 将每份 PDF 转换为标准化文档对象，保留文档级元数据、段落序号与正文文本。
- 在标准化结果基础上生成与数据库兼容的 paragraph 和 chunk 记录，并写入 PostgreSQL。
- 保证导入结果能够映射回 `document_id`、`chunk_id` 与段落范围，支撑后续在线检索与引用。
- 为失败文档、重复导入与数据校验提供最小必要的日志与校验输出。

**非目标：**

- 不在本次变更中实现在线 Python API、上传接口或后台任务系统。
- 不在本次变更中完成 Pinecone upsert 与 embedding 生成。
- 不修改现有 Hono 路由或在线检索编排逻辑。
- 不引入 OCR-first 的复杂解析链路，默认先处理可直接抽取文本的 PDF。

## 决策

### 决策一：采用“抽取 -> 标准化 -> 分块 -> 导入”四阶段离线流水线

先把 PDF 文本抽取与结构标准化从数据库导入阶段解耦，可以降低排错成本，也便于在导入前人工检查中间产物。建议每个阶段都产出明确的数据对象，而不是在一个脚本中直接从 PDF 写库。

备选方案：

- 单脚本一步到位写库：实现快，但难以定位问题，且不利于复跑与抽样核验。
- 先只输出 JSON 不写库：风险更低，但无法完成 MVP 所需的数据库落地闭环。

### 决策二：以文件名和标准化内容共同生成稳定 `document_id`

原始 PDF 文件名已包含案名、年份、法院与日期等线索，适合作为稳定标识输入的一部分。建议以规范化后的文件路径标识、核心元数据与文档标题组合生成 `document_id`，避免单纯依赖数据库自增或运行时 UUID，保证复跑时同一文档不会产生漂移。

备选方案：

- 每次导入生成随机 UUID：实现简单，但无法支持幂等导入和跨存储追踪。
- 完全依赖文件名哈希：简单稳定，但在文件名修订或人工清洗后可读性较差。

### 决策三：段落抽取优先保留法律文本结构，再按段落组装 chunk

法律检索需要段落级引用，因此离线流程不能使用纯固定长度切片。建议先识别并保留 paragraph 编号和顺序，再以 2 到 5 个相邻段落为一组生成 chunk；如果单段已足够长或具备完整法律意义，则允许单段成块。

备选方案：

- 直接按字符数或 token 数切块：实现简单，但会破坏段落边界与证据可追溯性。
- 完全依赖标题分节切块：结构更清晰，但很多判例并不具备稳定标题层级，容易得到过大块。

### 决策四：PostgreSQL 导入以现有 Drizzle schema 对应表为基线，使用 Python 显式映射字段

离线 Python 只负责向数据库写入与 schema 对齐的业务字段，不重新定义数据库结构。建议通过显式 SQL 或轻量映射层写入 `legal_documents`、`legal_document_paragraphs`、`legal_document_chunks`，并在导入前校验关键必填字段、唯一键与外键依赖。

备选方案：

- 在 Python 侧单独维护一套松散 JSON 模型：会增加与数据库模型漂移的风险。
- 让 TypeScript 再接管导入：会削弱 Python 离线处理闭环，也不符合本次“处理后直接入库”的目标。

## 风险与权衡

- [PDF 排版不规整导致段落识别错误] → 先实现基于规则的段落切分与编号提取，并保留原始抽取文本供回溯。
- [重复导入造成脏数据] → 使用稳定 `document_id` 与唯一约束配合 upsert 或导入前检查。
- [文档元数据抽取不完整] → 先从文件名补齐基础字段，正文解析只补充可可靠识别的信息。
- [chunk 与 paragraph 映射不一致] → 在生成 chunk 时显式记录起止段落号与 paragraph ID 列表，并增加校验步骤。
- [数据库写入与本地解析耦合过深] → 通过中间标准化结构隔离解析逻辑和入库逻辑，便于单独测试。
