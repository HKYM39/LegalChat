## 背景

当前系统的聊天主路径已经具备前端 `chat-first` 交互和后端 `/ask` 问答接口，但没有任何频率限制控制。由于问答链路会同时触发数据库检索、向量检索和模型生成，缺少限流会直接放大成本与误用风险。用户要求本次变更把限额做成环境变量，并且前后端都要生效，因此这不是单点实现，而是跨前端配置、前端状态层、后端配置和后端请求路径的联动变更。

约束如下：

- 后端必须继续保持 TypeScript + Hono 架构，不引入新的后端服务。
- 前端与后端的限额语义必须一致，避免展示值与实际拦截值不一致。
- 前端限流只能作为体验优化，不能替代后端的真实保护。
- MVP 范围内不引入账号体系，因此限流主体需要基于当前会话、客户端标识或等价匿名键实现。

## 目标 / 非目标

**目标：**

- 定义统一的聊天限流配置，至少支持每分钟 10 条、每天 100 条两个阈值，且可由 ENV 覆盖。
- 后端在 `/ask` 请求进入检索和模型调用前完成限流校验，超限时快速失败并返回结构化错误。
- 前端在 composer 提交前读取同样的 ENV 配置，对本地会话执行预检查，并在超限时给出明确提示。
- 明确前后端配置项、返回语义和测试覆盖点，使后续实现可直接落地。

**非目标：**

- 不在本次设计中引入用户登录、套餐、角色分级或多租户配额。
- 不在本次设计中实现跨设备、跨浏览器的强一致用户级配额同步。
- 不在本次设计中新增 Redis、消息队列或独立限流网关。
- 不把前端本地限流设计为安全边界，真实限制仍以后端为准。

## 关键决策

### 决策一：采用“双层限流”，后端强制执行，前端做预检查

后端必须作为最终裁决点，在 `/ask` 路由进入检索和模型调用前拦截超限请求；前端只做本地预检查和禁用提交，减少无效请求与糟糕反馈。

原因：

- 只有后端能够真正保护数据库、Pinecone 和模型成本。
- 只有前端做限流很容易被绕过。
- 只有后端做限流虽然安全，但用户会在提交后才看到错误，体验较差。

备选方案：

- 仅前端限流：实现简单，但没有保护作用，放弃。
- 仅后端限流：安全，但用户反馈滞后，不采用作为唯一方案。

### 决策二：限额配置拆成独立 ENV，前后端分别显式声明

使用分钟级与天级两个独立环境变量，例如 `CHAT_RATE_LIMIT_PER_MINUTE` 与 `CHAT_RATE_LIMIT_PER_DAY`。前后端都需要从各自配置入口显式读取，而不是把数值硬编码到界面或路由内部。

原因：

- 用户已明确要求把限制放在 ENV 中，且前后端都要有。
- 独立 ENV 便于本地、预发、生产环境分别调整。
- 显式配置比前端从后端动态拉取更适合当前 MVP 结构，改动更小。

备选方案：

- 后端单独持有 ENV，前端通过接口读取：一致性更强，但会新增配置接口或在现有接口中注入配置，超出这次最小提案范围。
- 代码常量硬编码：无法满足运维调整需求，放弃。

### 决策三：后端采用匿名主体键维护“分钟窗口 + 天窗口”计数

后端对每个聊天主体同时维护 60 秒窗口和自然日窗口两个计数器；请求进入 `/ask` 时先检查分钟额度，再检查天额度，任何一个超限都直接返回限流错误。主体键可基于客户端传入的稳定会话标识、cookie、IP 或其组合生成，但实现必须在设计落地时固定为单一规则并在文档中写明。

原因：

- 用户要求的限制本质上是两个独立窗口，需要并行判断。
- 自然日窗口更容易向用户解释“一天最多 100 条”。
- 在无登录体系的前提下，匿名主体键是当前 MVP 可接受的折中方案。

备选方案：

- 滑动 24 小时窗口：精度更高，但实现与用户心智更复杂。
- 仅分钟窗口：无法满足天级限制需求，放弃。

### 决策四：限流错误必须结构化，并包含用户可消费的信息

后端限流错误响应需要至少包含错误类型、命中的窗口类型、限制值和可读提示；前端本地预检查也需要产出相同语义的错误状态，避免出现“前端说还能发，后端说不行”的不透明体验。

原因：

- 聊天 UI 必须能区分普通失败与超限失败。
- 结构化错误更利于前端渲染、日志统计和后续扩展。

备选方案：

- 只返回纯文本报错：实现快，但难以稳定消费，不采用。

## 风险 / 权衡

- [匿名限流主体不够稳定] → 先在提案中允许使用会话标识或等价匿名键，实施时固定一种主策略，并在 README 与 ENV 说明中明确局限。
- [前后端 ENV 配置不一致] → 在文档和配置样例中使用同名或强对应命名，并要求前后端配置入口都提供默认值与启动时校验。
- [前端本地计数可被清理或绕过] → 明确前端仅为体验优化，所有真实拦截以后端为准。
- [天级窗口定义存在时区歧义] → 设计中约定以后端服务时区或 UTC 自然日为准，实施时在文档里固定。
- [限流逻辑污染核心问答代码] → 把限流判断下沉为独立配置与服务层，路由只负责调用和错误映射。

## 迁移计划

1. 新增前后端聊天限流 ENV 配置及样例值，默认满足“每分钟 10 条、每天 100 条”。
2. 后端接入限流服务与结构化错误，在 `/ask` 路径上线强制校验。
3. 前端读取同一组限额配置，在聊天提交前执行本地预检查并展示超限提示。
4. 补充前后端测试与运行说明，确认默认配置、禁用逻辑与超限响应都符合预期。
5. 如上线后发现误伤，可临时通过 ENV 提高阈值；如需快速回滚，可将后端阈值提高到足够大的值并移除前端禁用展示。

## 未决问题

- 后端匿名限流主体最终采用 IP、cookie 会话键，还是前端生成并持久化的客户端 ID，需要在实施前结合当前部署形态定稿。
- 天级窗口按服务端本地自然日还是 UTC 自然日结算，需要在实现阶段统一并写入说明。
- 是否需要把剩余额度一并暴露给前端，用于在 composer 中展示“剩余次数”，本次提案暂不强制。
