## ADDED Requirements

### Requirement: 聊天频率限制必须支持分钟级与天级 ENV 配置
系统 MUST 通过环境变量定义聊天频率限制，至少包含“每分钟最大聊天次数”和“每天最大聊天次数”两个独立阈值。前端与后端 MUST 都能读取对应配置，并使用一致的限额语义。

#### Scenario: 启动时读取默认聊天限额
- **WHEN** 前端或后端在未显式覆盖聊天限流环境变量的情况下启动
- **THEN** 系统使用约定的默认限额，并保证分钟级与天级阈值都可被显式覆盖

#### Scenario: 部署环境覆盖聊天限额
- **WHEN** 部署环境提供自定义聊天限流环境变量
- **THEN** 前端与后端都会读取覆盖后的阈值，而不是继续使用代码硬编码常量

### Requirement: 聊天频率限制必须同时覆盖分钟窗口与天窗口
系统 MUST 对同一聊天主体同时维护分钟级窗口与天级窗口的请求计数；任一窗口达到上限时，新的聊天请求 MUST 被判定为超限。

#### Scenario: 命中分钟级限制
- **WHEN** 同一聊天主体在 60 秒窗口内已达到分钟级聊天上限后再次发起聊天
- **THEN** 系统判定该请求超出分钟级额度，并阻止其继续进入问答主链路

#### Scenario: 命中天级限制
- **WHEN** 同一聊天主体在当天已达到天级聊天上限后再次发起聊天
- **THEN** 系统判定该请求超出天级额度，并阻止其继续进入问答主链路

### Requirement: 超限结果必须提供统一的结构化语义
系统 MUST 为聊天超限结果提供统一的结构化语义，至少包含错误类型、命中的窗口类型、限制值和面向用户的可读提示，以便前端和后端使用同一种表达处理超限状态。

#### Scenario: 后端返回结构化限流错误
- **WHEN** 后端判定聊天请求超限
- **THEN** 响应中包含可区分限流的错误类型、命中窗口和限制值，前端可以据此展示对应提示

#### Scenario: 前端生成本地超限状态
- **WHEN** 前端在提交前判定当前会话已达到本地聊天限额
- **THEN** 前端生成与后端语义一致的超限状态，用于禁用继续提交和展示提示信息
