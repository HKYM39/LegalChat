## ADDED Requirements

### Requirement: 前端必须基于 ENV 执行聊天限额预检查
系统 MUST 在聊天输入与提交主路径中读取聊天限流环境变量，并在用户提交前对当前前端会话执行分钟级与天级限额预检查。当前端判定已超限时，MUST 阻止继续提交。

#### Scenario: 前端会话命中分钟级限额
- **WHEN** 用户在当前前端会话的 60 秒窗口内已经达到分钟级聊天上限后再次提交问题
- **THEN** 前端阻止本次提交，并保持输入框处于可解释的受限状态

#### Scenario: 前端会话命中天级限额
- **WHEN** 用户在当前前端会话当天已经达到天级聊天上限后再次提交问题
- **THEN** 前端阻止本次提交，并明确提示当天聊天额度已用尽

### Requirement: 前端必须明确展示聊天超限反馈
系统 MUST 区分聊天超限与普通请求失败；无论是本地预检查触发还是后端返回限流错误，界面都 MUST 提供清晰、可恢复理解的提示，而不是显示为通用异常。

#### Scenario: 本地预检查阻止提交
- **WHEN** 前端在提交前判定会话已超出聊天额度
- **THEN** composer 展示明确的聊天限额提示，并且不发起 `/ask` 请求

#### Scenario: 后端返回限流错误
- **WHEN** 前端提交请求后收到后端返回的聊天限流错误
- **THEN** 系统将错误渲染为聊天额度提示，而不是通用“请求失败”消息
