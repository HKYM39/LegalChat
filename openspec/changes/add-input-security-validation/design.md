## 背景

当前聊天主链路只依赖请求结构解析与业务 schema 校验，尚未建立面向安全风险的输入边界。前端会将用户原始文本直接发送给 `POST /ask`，后端在进入检索、日志与模型调用前也没有统一的高风险载荷拒绝层，这会让明显异常输入进入不必要的处理路径。

这次变更同时影响 `apps/web`、`apps/api` 与共享类型层，属于跨模块安全变更。仓库约束要求在线能力保持“检索优先、可追溯、保守回答”，因此输入安全校验必须发生在问答业务主链路之前，且不能让模型成为安全判定器。

## 目标 / 非目标

**目标：**

- 建立前后端一致的聊天输入安全边界，阻止明显非法或攻击型 payload 进入后端问答主链路。
- 让前端在本地提交流程中尽早拦截，减少无效请求与错误噪声。
- 让后端作为权威校验层，在任何客户端绕过前端时仍可主动拒绝。
- 提供统一的结构化错误语义，便于前端展示、测试断言与后续扩展。
- 保持对正常法律研究问题的通过率，不把通用长度校验、空值校验与安全拒绝混为一类。

**非目标：**

- 不实现通用 WAF、IP 封禁、验证码、账号风控或行为评分系统。
- 不覆盖所有未知攻击变体，也不承诺替代基础设施层防护。
- 不把输入内容改写为“安全版本”后继续发送；命中高风险规则时直接拒绝。
- 不新增鉴权、租户隔离、后台审核或上传文件扫描能力。

## 关键决策

### 决策一：建立共享的“输入安全判定器”，由后端权威执行，前端复用相同规则

采用共享模块定义 `validateUserInputSecurity()` 一类纯函数，输入原始字符串，输出 `allowed`、`reasonCode`、`message`、`matchedSignals` 等结构。前端在提交前调用同一套规则进行预检，后端在 `/ask` 路由解析完成后再次执行，并以服务端结果为准。

原因：

- 规则集中维护，避免前后端产生漂移。
- 前端可以减少明显无效请求，后端仍然具备不可绕过的兜底能力。
- 纯函数更适合单元测试和边界样例覆盖。

备选方案：

- 仅前端校验：无法防止直接调用后端接口，放弃。
- 仅后端校验：安全上可行，但会增加无效请求、损失即时反馈，放弃。
- 引入独立安全服务：当前 MVP 复杂度过高，放弃。

### 决策二：采用“规范化 + 规则信号”模型，而不是单纯关键词黑名单

判定流程先做输入规范化，再按规则信号分类拒绝。规范化包括去除前后空白、统一换行、检测不可见控制字符、统计长度。规则信号聚焦“明显不应进入法律问答”的 payload，例如：

- 超过约定最大长度或为空白内容
- 非文本控制字符、二进制片段或异常编码形态
- 明显的 HTTP 请求走私/协议探测载荷，例如请求行、Host 头、原始 header 块
- 明显的脚本注入片段，例如 `script` 标签、事件处理属性、`javascript:` / `data:` 伪协议
- 明显的 SQL/命令/模板注入探测片段、路径穿越或本地文件探测片段

原因：

- 单纯黑名单容易被绕过，也容易因词面命中产生误杀。
- 以结构信号为主更贴近“网络攻击非法内容”的边界。
- 先做规范化有利于前后端使用一致语义。

备选方案：

- 只做 Zod 字段长度校验：覆盖不到攻击型 payload，放弃。
- 让 LLM 判断是否恶意：不稳定、成本高、不可追溯，放弃。

### 决策三：后端拒绝响应使用结构化错误码，并且不让请求进入检索、模型和主日志链路

后端在命中安全拒绝时直接返回结构化错误，包含固定错误类型、原因码和用户可读提示。该请求不得进入检索、向量查询、模型生成与正常问答日志记录；如需观测，只记录最小必要的安全事件摘要，不存储完整原始 payload。

原因：

- 符合“先拒绝、后终止”的安全边界。
- 减少恶意载荷在日志、提示词或下游服务中扩散。
- 前端可以稳定地区分安全拒绝与普通失败。

备选方案：

- 继续进入主链路但在 limitations 中提示：会让恶意输入接触检索与模型，放弃。
- 记录完整 payload 便于排查：会增加敏感内容扩散面，放弃。

### 决策四：前端对安全拒绝做明确提示，但不尝试自动修复内容

前端命中本地规则时，禁止提交并展示解释性错误；若后端返回同类错误，则将其渲染为同一种安全提示。前端不自动裁剪、转义或删除可疑片段后重试，避免把原本应拒绝的内容“洗白”后继续提交。

原因：

- 自动修复容易制造不可预期语义变化。
- 对安全边界而言，拒绝比静默篡改更透明。

备选方案：

- 自动 sanitize 后提交：行为不可解释，且可能掩盖攻击意图，放弃。

## 风险 / 权衡

- [误杀合法研究语句] → 规则优先匹配结构性攻击片段，避免仅凭单个敏感词拒绝；为常见法律研究语句补充回归样例。
- [前后端规则漂移] → 规则集中在共享模块，前端与后端直接复用同一个判定器与原因码枚举。
- [新型 payload 漏检] → 先覆盖明显高风险类别，并保留可扩展原因码与测试样例库，后续增量补强。
- [错误提示泄露过多判定细节] → 面向用户仅暴露抽象原因与修改建议，不返回具体正则或检测细节。
- [日志与监控缺失] → 为安全拒绝补充最小必要事件计数或摘要记录，但不持久化完整恶意原文。

## 迁移计划

1. 在共享层新增输入安全判定类型、原因码与纯函数校验器。
2. 在前端聊天提交路径接入预检，阻止命中规则的内容发起 `/ask`。
3. 在后端 `/ask` 路由或等价入口接入权威校验，命中后直接返回结构化错误。
4. 为正常请求、前端拒绝、后端拒绝分别补充单元测试与接口测试。
5. 部署时无需数据迁移；若发现误杀，可通过回滚对应代码版本恢复到旧行为。

## 未决问题

- 安全拒绝是否需要单独的轻量审计计数存储，以及存储位置是应用日志还是数据库事件表。
- 最大输入长度阈值是沿用现有 ask schema，还是单独为安全校验定义更严格上限。
- 是否需要将同一套校验同步用于 `GET /search`，本次提案先聚焦聊天主路径。
