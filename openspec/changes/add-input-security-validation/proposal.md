## 背景

当前聊天输入会直接从前端流向后端问答接口，缺少一套明确的输入安全边界。若用户提交超长内容、控制字符、脚本片段、协议探测载荷或明显的攻击型 payload，系统既可能放大后端处理风险，也会让日志、检索与模型调用暴露在不必要的异常输入之下，因此需要在前后端同时建立主动拒绝机制。

## 变更内容

- 新增统一的输入安全校验能力，定义聊天输入允许范围、拒绝条件、错误语义与前后端一致的判定原则。
- 前端在输入与提交阶段执行本地安全校验，拦截明显非法内容，禁止将命中规则的 payload 发送给后端。
- 后端在 `POST /ask` 入口强制执行服务端校验，对非法或高风险输入直接拒绝，并返回可识别的结构化错误。
- 补充共享类型、配置说明与测试要求，确保前端校验失败、后端兜底拒绝、正常法律问题通过这三类路径都有明确行为约定。

## 能力范围

### 新增能力
- `input-security-validation`: 定义用户输入的安全边界、前后端统一校验规则、拒绝响应格式，以及对明显网络攻击或异常 payload 的处理原则。

### 修改能力
- `chat-first-legal-rag-frontend`: 聊天输入框与提交流程需要在发送前执行输入安全校验，并对被拒绝内容展示明确错误提示。
- `online-legal-rag-backend`: 在线问答接口需要在业务处理前执行输入安全校验，对不符合规则的请求返回结构化拒绝结果，不进入检索、模型调用与日志主链路。

## 影响范围

- 影响 `apps/web` 的聊天输入组件、状态管理、错误提示与 API 调用前置校验流程。
- 影响 `apps/api` 的 `/ask` 路由、请求解析、错误响应与输入校验中间逻辑。
- 影响 `packages/shared` 或等价共享模块中的请求 schema、校验结果类型与错误码约定。
- 影响前后端测试覆盖，新增安全输入、非法载荷、边界长度与回归行为验证。
